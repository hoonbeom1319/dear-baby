# dear-baby V2 — 제품 요구사항 문서 (PRD)

> **버전:** 1.0 (MVP 기준)
> **상태:** 기획 완료, 개발 착수 전
> **작성 기준:** 1~7단계 + 개념 데이터 모델 합의 완료
> **만드는 사람:** 1인 풀스택 (Vercel + Supabase), Claude Code/Cursor 활용
> **한 줄 정의:** 아이와 간 장소를 날짜별·사진과 함께 지도에 기록하여, 공간 위에 추억을 축적하고 되새기는 "혼자(가족) 보는 디지털 추억 지도"

---

## 1. 배경과 문제 정의

### 1.1 V1의 한계 (왜 V2인가)

V1(dear-baby)은 아이와 갈 만한 장소를 운영자가 직접 큐레이션해 보여주는 탐색 도구였다. 실패의 직접 원인은 기술이 아니라 **운영 부담**이었다. 데이터(정확성·최신성·커버리지)가 운영자 한 사람의 노동에 전부 묶여 있어 지속 불가능했다. 이 위험은 V1 기획 당시에도 식별되었고 현실이 되었다.

V2의 발상 전환: **운영자가 데이터를 채우는 구조를 버리고, 사용자가 자기 추억을 기록하면서 데이터가 자연히 쌓이는 구조로 전환한다.**

### 1.2 해결하려는 문제

아이와의 추억을 인스타·블로그·네이버 지도에 기록하지만, 이들은 모두 **시간순**이라 "공간 위에 아이와의 여정이 쌓인 모습"을 한눈에 볼 수 없다. 지도 즐겨찾기는 추억이 아니라 북마크다. 즉 **공간 위에 축적되는 추억, 그리고 그것을 되새기는 경험**이 부재하다.

### 1.3 해결 방식과 핵심 가치

아이와 간 장소를 날짜별로, 사진과 함께 지도에 기록한다. 시간이 쌓이면 지도가 추억으로 가득 차고 "우리 진짜 많이 다녔구나" 하는 되새김의 감동을 준다.

핵심 가치 세 가지:
- **공간적 축적** — 인스타·블로그가 주지 못하는 것
- **되새김** — 지도 즐겨찾기가 주지 못하는 것
- **아이 중심** — 내 계정의 일부가 아닌, 우리 아이와의 여정

### 1.4 제품 정체성

혼자(가족) 보는 디지털 추억 지도. **공유 없음(MVP 기준).** dear-baby를 대체하는 새 서비스이며, 장소 탐색은 데이터가 쌓이면 따라오는 부가 가치일 뿐 주 기능이 아니다.

## 2. 목표와 성공 지표

### 2.1 성공의 정의

운영자가 택한 성공상은 **"내 가족 + 진짜로 쓰는 소수의 가족이 꾸준히 돌아오는 도구"** (성장보다 깊이). 따라서 획득·성장 지표(신규 설치 수, 가입 전환율)는 북극성에서 제외하며, **유지와 깊이**가 성공의 척도다.

여기에 운영 가드레일을 성공 조건으로 명시한다: **사용자당 운영자 개입 시간 ≈ 0.** V1을 죽인 직접 원인이므로, 사용자 지표가 좋아도 운영 부담이 생기면 실패로 본다.

### 2.2 V2의 핵심 베팅

V2는 공유·소셜 루프를 의도적으로 전부 제거했다. 그 대가로 재방문을 만드는 가장 강력한 엔진(소셜 루프)도 사라졌다. 따라서 V2는 오직 두 내재 동기에만 기댄다:
- **기록 습관** ("다녀왔으니 남겨야지") — 데이터를 쌓는 엔진
- **되새김 보상** ("지도를 열어 우리가 쌓은 걸 본다") — 가치가 전달되는 순간

**핵심 베팅:** "소셜 루프 없이, 내재 동기만으로 사람이 계속 돌아온다." 2단계 지표의 임무는 성공 측정이 아니라 **이 베팅이 진짜인지를 빠르고 싸게 검증하는 것**이다.

### 2.3 측정 도구 = 가족별 추적 + 셀프 베타

목표가 "소수의 가족"(5~20명 규모)이므로 퍼센트 기반 지표(리텐션 70% 등)는 통계적으로 무의미하다. 대신 **가족 한 명 한 명을 이름 붙여 추적하는 표 + 정성 인터뷰**를 측정 도구로 삼는다. 운영자 본인 가족이 첫 코호트(셀프 베타)이며, 이는 "그냥 테스트"가 아니라 **지표값을 발견하는 정식 측정 기간**이다.

### 2.4 북극성 지표 (NSM)

**활성 추억 가족 수** — 최근 60일 안에 ① 새 핀을 1개 이상 남기고 *동시에* ② 기록과는 별개의 세션에서 지도를 열어 둘러본 가족의 수.

- 두 행동을 동시에 요구하는 이유: 기록만 하면 엔진은 도는데 가치 미전달, 보기만 하면 가치는 받는데 엔진 정지. 둘 다 충족돼야 "루프가 닫혔다".

### 2.5 보조 지표 (항목 확정 / 임계값은 셀프 베타로 발견)

| 구분 | 지표 | 무엇을 답하는가 |
|---|---|---|
| 엔진 | 월간 기록 지속 | 외출 → 기록 습관이 붙었나 |
| 엔진 | 가족당 핀 누적 속도 (활성 월 기준 중앙값) | 계곡을 향해 가고 있나 |
| 엔진 | 멀티-장소 기록 비율 (한 세션 2곳 이상) | "하루 여러 장소" 마찰이 한 곳에서 포기시키나 |
| 가치 | 순수 되새김 세션 수 (핀 안 만들고 지도만 열어본 세션) | 되새김이 실제로 일어나나 (a 베팅의 핵심) |
| 가치 | 첫 되새김까지의 핀 수 (가족별) | 계곡 바닥이 핀 몇 개에 있나 |
| 계곡 | 계곡 통과율 + 통과까지 기간/핀 수 | 첫 기록 가족 중 몇이 "추억 가득한 지도"에 도달했나 |
| 가드레일 | 월 운영자 개입 시간 (운영 한정, 개발 제외) | V1처럼 운영 부담으로 죽고 있나 (목표 ≈ 0) |

**계곡 통과의 조작적 정의:** "순수 되새김 세션을 1회 이상 했고 핀 N개 이상 보유"한 시점에 그 가족은 계곡을 건넌 것으로 본다. N은 가설값(10~20)이되, 목표가 아니라 **첫 코호트에서 발견할 값**. '첫 되새김까지의 핀 수'가 쌓이면 N의 실제 위치를 데이터가 알려준다.

### 2.6 측정 창과 임계값: 셀프 베타로 확정 (열어둠)

측정 창(60일/30일)과 계곡 임계치 N은 운영자 셀프 베타 후 확정한다. 다만 **측정 대상 두 사건은 지금 고정**하며, 첫 버전부터 로그로 남겨야 한다:
- 핀을 만든 사건 (가족, 시각)
- 기록 없이 지도를 열어 본 사건 = 순수 되새김 세션 (가족, 시각)

→ 되새김 세션은 의도적으로 구분해 남기지 않으면 "앱 열었다"에 묻혀 사라진다. 데이터 모델의 Event 엔터티가 이를 담당한다.

### 2.7 감시 대상 위험

a 결정("앱은 가만히, 사용자가 스스로 지도를 연다")에 따라, **"앱이 가만히 있는데 사람이 스스로 지도를 다시 열까?"** 가 제품 전체에서 가장 얇은 가정이 되었다. 사람들이 기록만 하고 안 들여다보면 가치는 영영 안 닿고 루프는 안 닫혀 "쓰기 귀찮은 사진첩"이 된다.

연기 감지기: **순수 되새김 세션 수**와 **첫 되새김까지의 핀 수.** 첫 코호트에서 이 둘이 죽어 있으면 a 베팅이 틀린 것이고, 그때 (b)형 넛지("1년 전 오늘, 여기 갔었네요")가 Phase 1.5의 1순위 처방으로 올라온다. (MVP엔 미포함)

추가 감시 가정: **"기록은 외출에 묶여 자연 증가한다"는 낙관이 약화되었다.** 실제 기록 트리거는 외출이 아니라 "회상하며 올리고 싶은 순간"이며(인스타·블로그처럼 지난 일을 생각날 때 올림), 이 순간을 유발하는 장치가 MVP에 없다. 셀프 베타의 1순위 관찰 대상.

## 3. 타깃 사용자와 사용 시나리오

### 3.1 핵심 사용자

운영자 본인 가족을 첫 사용자로, 이후 **운영자의 실제 지인(친구·동료 부모)에게 자연히 번지는** 그림. 낯선 사용자 모객이 아니라 신뢰가 이미 깔린 사람들. ("(가) 지인 기반에 가까운 (다) 자연 확산")

- **유리한 점:** 지인 기반이라 첫인상 압박이 적다. 거친 MVP도 봐준다. → MVP를 더 작게 내도 된다.
- **함정:** 운영자는 제품 설계자이자 개발자라 기록 흐름이 번거로워도 머릿속 모델로 30초 만에 해낸다. 그래서 "기록 마찰"을 과소평가하기 쉽다. **진짜 검증은 옆에서 두 번째로 써보는 지인 부모 1명에게서 나온다.** → 셀프 베타 다음에 지인 1~2가족을 의식적으로 일찍 끌어들일 것.

> **참고:** 인구통계(나이·직업)는 제품에 거의 영향을 주지 않는다. 중요한 것은 **기록하는 순간의 맥락**이다.

### 3.2 기록의 맥락 (화면 설계의 근거)

- **기록 시점:** 귀가 후 몰아서, 또는 지난주 것을 생각날 때. (현장에서 실시간 아님)
- **사진 출처:** 휴대폰 갤러리에서 고름. (그 자리에서 앱 카메라로 찍는 게 아님)
- **묶음 단위:** 하루.
- **하루 여러 장소가 핵심:** 예) 하남 스타필드 → 다산 생태공원 → 카페. 한 번의 기록 세션에서 여러 장소를 가볍게 다룰 수 있어야 한다.

이 맥락이 잠그는 설계: 현장형 기능(GPS 실시간 핀, 체크인) 불필요. 날짜를 사용자가 고를 수 있어야 함(지난 것 올리는 게 정상). 갤러리 사진의 EXIF(촬영시각·GPS)를 읽어 자동 채우기 가능 → 마찰 극적 감소.

### 3.3 핵심 시나리오 — 기록하기 (30초~2분)

> 금요일 밤, 아이 재우고 소파. 지난 주말 나들이를 아직 안 올렸다는 게 생각났다. 앱을 연다. "기록 추가"를 누르고 갤러리에서 그날 사진 8장을 고른다. → 앱이 EXIF를 읽어 *"10/12 토요일"* 과 함께 **장소 3개(스타필드·생태공원·카페)로 자동 그룹핑**하고 사진을 각각 매핑해 보여준다. 사용자는 장소 이름만 한 번씩 확인하고, 멘트는 생략(옵션). 저장. → 지도에 그날의 동선 위로 핀 3개가 나란히 박힌다. **2분 안에 끝.**

### 3.4 핵심 시나리오 — 되새기기

> 어느 저녁, 문득 앱을 연다. 핀이 가득 박힌 지도가 펼쳐진다. "우리 이만큼 다녔구나." 자주 갔던 키즈카페 핀을 누른다. 거기 갔던 모든 방문이 시간순으로 펼쳐진다 — 작년 봄, 여름, 올겨울. 그날의 사진과 짧은 멘트를 다시 본다.

## 4. 기능 요구사항

> 기능 명세는 "사용자가 X하면 시스템은 Y한다" 형태로 기술하며, 엣지 케이스를 함께 적는다. 우선순위: **P0**(MVP 필수) / **P1**(Phase 1.5) / **P2**(Phase 2).

### 4.1 기록하는 길 (P0)

#### F-1. 하루치 사진 일괄 업로드 (P0)
- 사용자가 "기록 추가"를 누르면, 시스템은 갤러리에서 여러 장의 사진을 선택하게 한다.
- 사용자가 사진 선택을 완료하면, 시스템은 각 사진의 EXIF(촬영시각, GPS)를 **브라우저에서** 읽는다. (서버 업로드 전)
- 엣지: EXIF가 없는 사진, GPS만 없는 사진, 촬영시각만 없는 사진이 섞일 수 있다 → 각각 처리(F-2, F-3).

#### F-2. EXIF 기반 자동 그룹핑 + 장소 제안 (P0)
- 시스템은 GPS가 있는 사진을 **위치 근접성 기준으로 장소별로 그룹핑**한다.
- 각 그룹에 대해 카카오맵 역지오코딩으로 **장소명을 제안**한다.
- 시스템은 촬영시각 중 대표값을 **방문 날짜로 자동 제안**한다.
- 사용자는 제안을 확인·수정·승인한다. **자동은 "첫 제안"까지만 책임지고, 최종 확정·수정은 항상 사람이 한다.**
- 엣지: 대형 건물·실내는 좌표→장소 매칭이 부정확하거나 후보가 여럿일 수 있다 → 사용자가 수정 가능해야 함. (역지오코딩 정확도는 미검증 가정 — §8 참조)

#### F-3. 위치 없는 사진 처리 (P0, 메인 흐름의 일부)
- GPS가 없는 사진은 시스템이 **"미분류" 영역에 따로 모은다.**
- 사용자는 미분류 사진을 특정 장소 그룹에 배치하거나, 새 장소를 만들어 배치한다.
- **중요:** "위치 없는 경우"는 예외가 아니라 흔한 경우일 수 있다(메신저·SNS 경유 사진은 GPS 제거됨, 위치 태그 off 사용자 등). 위치 있는 흐름과 **동등한 무게**로 설계한다.

#### F-4. 하루치 편집기 — 단일 편집 모델 (P0) ★ 가장 중요한 화면
- 자동 분석 결과(장소별 그룹 + 미분류)가 **"이미 채워진 상태"** 로 제시된다.
- 사용자는 그 위에서 자유롭게 편집한다. 지원 동작:
  - 장소 추가 (빈 장소 생성, 이름·핀 지정) — "할머니 집" 등 카카오맵에 없는 곳 포함
  - 장소 삭제
  - 장소 이름·핀 위치 수정
  - 사진을 장소에 넣기 / 다른 장소로 옮기기 / 빼기
  - 사진 삭제
  - 방문 날짜 수정 (자동값 위에서 변경)
  - 멘트 입력 (선택)
- **설계 원칙:** 자동 채우기와 수동 편집은 별개 기능이 아니라 **하나의 편집기**다. 자동 EXIF 분석은 그 편집기를 "빈 칸"이 아니라 "이미 채워진 상태"에서 시작시킬 뿐이다.
- 사용자가 "저장"하면, 시스템은 해당 장소·방문·사진을 생성하고 지도에 핀으로 표시한다.
- **MVP 제외:** 장소 "쪼개기"/"합치기" 전용 기능. → 장소 추가 + 사진 이동 + 장소 삭제의 조합으로 동일 결과 도달 가능하므로, 별도 버튼은 만들지 않는다(편의 단축키 1개가 빠질 뿐, 결과는 100% 도달).

#### F-5. 사진 없이 장소만 기록 (P0, 작은 샛길)
- 사용자가 사진 없이 장소만 남기고 싶을 때, 메인 흐름에서 "장소 직접 추가"로 빠질 수 있다.
- 전체의 소수 케이스로 가정. (입구는 어디까지나 사진)

### 4.2 되새기는 길 (P0)

#### F-6. 지도 홈 — 핀이 박힌 전체 지도 (P0) ★ 되새김의 메인 무대
- 사용자가 앱(또는 지도 탭)을 열면, 시스템은 **줌아웃된 전체 지도에 모든 핀을 표시**한다. "공간적 축적"이 첫인상.
- a 결정에 따라, 이 화면 자체가 되새김의 감동을 줘야 한다(별도 넛지 없음).
- 빈/적은 핀 상태(계곡 구간)는 "빈약함"이 아니라 "채워가는 여백"으로 느껴지게 한다. → §7 디자인 과제.

#### F-7. 장소 상세 — 방문 시간순 (P0)
- 사용자가 핀을 누르면, 시스템은 **그 장소의 모든 방문을 시간순으로** 보여준다. ("여기 3번 왔었네")
- 각 방문에는 날짜, 사진들, 멘트(있으면)가 딸린다.
- 사진·멘트의 소속은 **장소가 아니라 방문**이다(같은 장소라도 방문마다 사진·멘트가 다름).

### 4.3 계정 (P0)

#### F-8. 로그인 (P0)
- 개인 계정 기반. 1계정 = 1가족의 추억 지도.
- **로그인 시점:** 첫 기록을 시도하는 순간에 로그인을 받는다(앱 진입 즉시 아님). 가치를 만들려는 순간에 인증 → 계곡 입구 마찰 최소화.
- **요구사항:** 로그인 전에 고른 사진·편집 내용은 로그인 후에도 보존되어야 한다.
- **폴백 허용:** 위 보존 구현이 1인 개발에 과한 부담이면, 개발 단계에서 "앱 진입 시 로그인"(가)으로 후퇴 가능.

### 4.4 측정 (P0)

#### F-9. 이벤트 로깅 (P0)
- 시스템은 다음 두 사건을 첫 버전부터 로그로 남긴다: ① 핀 생성, ② 순수 되새김 세션(기록 없이 지도 열람).
- 측정값(60일/N)은 나중에 정하더라도, 측정 대상은 지금부터 쌓아야 한다.

### 4.5 Phase 1.5 / Phase 2 (MVP 제외, 예약)

| 기능 | 단계 | 비고 |
|---|---|---|
| 히트맵 (자주 간 곳을 뜨겁게, 가끔 간 곳을 차갑게) | P1 | 핀이 충분히 쌓인 계곡 이후에야 의미. 데이터(방문 수)는 MVP에서 이미 갖춰지므로 추가 데이터 불필요. 표현은 그때 디자인이 풀 것 |
| (b)형 넛지 ("1년 전 오늘 여기 갔었네요") | P1 | a 베팅이 틀렸을 때(되새김 세션이 안 일어날 때)의 1순위 처방 |
| 기존 장소 재방문 자동 감지·합치기 제안 | P1 | MVP는 매번 새 핀 + 사용자가 정리. 셀프 베타에서 중복이 거슬리면 승격 |
| 공유 기능 전체 (남의 추억 보기, 공개/비공개, 신고) | P2 | 아동 프라이버시 위험 회피 + 운영 부담 회피 |
| 장소 탐색을 주 기능화 | P2 | 데이터 쌓인 뒤 부가 가치 |

## 5. 데이터 모델 (개념)

> 물리 스키마(컬럼 타입, 인덱스, 정규화, 외래키 제약 등)는 **개발 단계 몫**으로, 여기서는 다루지 않는다.

### 5.1 핵심 엔터티

| 엔터티 | 정체 | 주요 속성(개념) |
|---|---|---|
| **User** | 개인 계정, 모든 데이터의 주인 | 계정 식별자, 로그인 정보, 가입일 |
| **Place** | 지도의 핀 하나 (장소) | 이름, 위경도, 출처(카카오 POI / 자유 입력), 카카오 장소ID(옵션) |
| **Visit** | "어느 날 이 장소에 갔다" (= 하루치 한 장소) | 방문 날짜, 멘트(선택), 생성 시각 |
| **Photo** | 방문에 딸린 사진 | 이미지 저장 위치, 촬영시각 |
| **Event** | 측정용 행동 로그 | 사용자, 이벤트 종류(핀 생성 / 되새김 세션), 발생 시각 |

### 5.2 관계 (텍스트 ERD)

```
User (1) ──< (N) Place (1) ──< (N) Visit (1) ──< (N) Photo
   │
   └──< (N) Event
```

- **User 1 : N Place** — 한 사용자의 여러 장소
- **Place 1 : N Visit** — 한 장소에 여러 방문이 쌓임 ← **모델의 척추** (되새기기 §4.2 F-7의 근거)
- **Visit 1 : N Photo** — 한 방문에 여러 사진
- **User 1 : N Event** — 행동 로그
- **사진·멘트의 소속 = Visit** (Place가 아님). 같은 장소라도 방문마다 사진·멘트가 다르므로, 이 층이 있어야 "여러 번 방문"이 표현된다.

### 5.3 주요 결정 사항

- **사진의 원본 GPS는 저장하지 않는다.** 자동 그룹핑(장소 찾기)에만 쓰고 버린다. 일단 장소가 정해지면 진실의 원천은 Place의 핀 좌표지 사진이 아니므로, 영구 보관 실익이 MVP엔 없다. 아동 프라이버시 회피 방향과도 일관. (촬영시각은 방문 날짜 자동 채우기에 쓰이므로 유지하되, 영구 보관 필요 여부는 개발 단계 판단)
- **장소 = 카카오맵 POI 또는 자유 입력.** 자유 입력 허용으로 "할머니 집·동네 놀이터" 같은 곳도 기록 가능. 대신 "같은 곳을 다르게 적은" 중복 가능성 존재 → 재방문 처리는 P1로 예약(§4.5).

### 5.4 한 번의 기록 세션이 만드는 것

사진 8장 업로드 → EXIF 분석 → Place 3개 + 각 Place 아래 Visit 1개(그날 날짜) + 각 Visit에 Photo 매핑. 즉 한 세션이 Place·Visit·Photo를 동시에 생성한다.

## 6. 사용자 흐름과 화면

> 화면의 UI/UX 형태(레이아웃, 컴포넌트 모양)는 **디자인 단계(Claude Design)에서 확정**한다. 본 문서는 각 화면이 충족해야 할 **요구사항(제약)**을 명시한다.

### 6.1 시나리오 A — 기록하기

| 화면 | 역할 | 요구사항 |
|---|---|---|
| **A-1. 지도(홈)** | 진입점 | 핀 박힌 지도 + "기록 추가" 진입 |
| **A-2. 사진 선택** | 갤러리에서 하루치 다중 선택 | 선택 완료 시 EXIF 분석 시작 |
| **A-3. 하루치 편집기** | 자동 분석 결과 위에서 편집 | §6.3 별도 명시 |
| **A-4. 저장 완료 → 지도 복귀** | 즉각적 되새김 씨앗 | 방금 박힌 핀을 지도에서 바로 보여줌 |

로그인은 A-2~A-3 사이(첫 기록 시도 시점)에서 받는다(F-8).

### 6.2 시나리오 B — 되새기기

| 화면 | 역할 | 요구사항 |
|---|---|---|
| **B-1. 지도(홈)** | 되새김 메인 무대 (A-1과 동일) | 핀 박힌 전체 지도 |
| **B-2. 장소 상세** | 방문 시간순 열람 | 핀 클릭 시 해당 장소 모든 방문을 시간순으로. 방문별 사진·멘트. 사진 표현 형태는 디자인이 확정(운영자 선호: 방문별 대표 1장 → 펼침) |

> B의 화면 수가 적은 것은 정상이자 좋은 신호다. a 결정에 따라 되새김엔 별도 기능이 없어야 한다. 여기에 "되새김 전용 기능"을 더하고 싶으면 a 결정을 흔드는 것이므로 멈추고 P1 넛지 후보로 보낸다.

### 6.3 A-3 하루치 편집기 — 디자인에 넘기는 제약

UI 형태는 디자인 단계에서 확정하되, 어떤 형태든 다음을 반드시 충족한다:
- 자동 분석 결과(장소별 그룹 + 미분류)가 "이미 채워진 상태"로 보일 것
- 사진을 장소 간에 옮길 수 있을 것
- 장소 추가/삭제/이름·핀 수정이 가능할 것
- 미분류 사진을 장소에 배치할 수 있을 것
- 멘트는 눈에 안 띄어도 되지만(선택), 원하면 닿을 수 있을 것
- **지친 부모가 30초~1분 안에 끝낼 수 있을 것** ← 디자인이 풀어야 할 핵심 과제

### 6.4 시나리오 C — 처음 시작 (빈 지도의 계곡 입구)

| 화면 | 역할 | 요구사항 |
|---|---|---|
| **C-1. 빈 지도 + 첫 기록 유도** | 계곡 입구 | 핀 0개 상태에서 첫 기록으로 유도. 기능은 거의 없고 대부분 디자인·카피의 문제 |

기획이 못 박는 제약: **첫 기록까지의 마찰을 최소화할 것.** 긴 회원가입·다단계 튜토리얼은 계곡 입구에서 사용자를 죽인다. 이상적으로 "앱 열자마자 거의 바로 첫 사진 올리기"에 닿게 한다.

### 6.5 5단계에서 개발로 넘긴 메모

- **기존 장소 재방문 처리:** MVP는 매번 새 핀을 만들고, 중복이 생기면 사용자가 편집 동작으로 직접 정리한다(자동 감지·합치기 제안은 P1). 단골 장소 중복이 셀프 베타에서 거슬리는지 관찰 후 P1 승격 판단.

## 7. 디자인 방향

> 디자인 단계(Claude Design)의 출발점. 상세 UI는 디자인이 확정.

- **톤:** 따뜻하고 사적인 "우리 가족의 추억 지도". 화려함보다 정서. 아이 중심의 다정함.
- **모바일 우선.** 기록(귀가 후 소파에서)도 되새김(문득 열어봄)도 모바일 맥락. 데스크톱은 후순위.
- **레퍼런스 (성격별):**
  - 지도 위 기록·핀 경험 → 구글 타임라인 / 네이버·카카오 지도의 저장 경험 (단, 이들은 "북마크"에 그침 — 우리는 "추억"으로 차별화)
  - 사진 묶음·앨범 정서 → 인스타그램 / 폰 기본 사진 앱의 "추억" 기능 (단, 이들은 시간순 — 우리는 공간순)
  - 따뜻한 기록 톤 → 육아 기록·다이어리 계열 앱
- **다크모드:** MVP 필수 아님. 디자인 판단에 맡김.
- **핵심 디자인 과제:** 핀이 적은 초기 상태(계곡 구간)를 "빈약함"이 아니라 **"채워가는 여백"**으로 느끼게 할 것. 같은 핀 3개라도 프레이밍에 따라 "초라함"과 "여백의 설렘"으로 갈린다. 운영자가 셀프 베타에서 가장 먼저 자기 감정을 체크할 지점("내 핀 3개 지도를 봤을 때 설레나, 초라하나?").

## 8. 비기능 요구사항 / 기술 제약

- **스택:** Vercel + Supabase (운영자 숙련). 클라이언트 중심.
- **EXIF 읽기:** 브라우저에서 처리(서버 업로드 전). 경량 EXIF 라이브러리(GPS·촬영시각만 읽으면 ~4KB 수준)로 충분. 서버 불필요. — 검증 완료.
- **카카오 지도 연동:** 두 용도 — ① 장소 검색(사용자 수동 지정), ② 역지오코딩(좌표→장소명, 자동 제안). V1에서 연동 경험 있음.
- **운영 부담 ≈ 0:** 운영이 필요한 기능(공유·신고·모니터링)을 MVP에서 만들지 않는 것으로 구조적으로 방어. 이 원칙을 일정에서도 지킬 것.

### 8.1 개발 위험 (착수 시 유의)

1. **A-3 편집기가 일정의 병목.** 다른 화면은 단순하나 A-3(그룹핑·사진 이동·장소 편집)에 시간이 몰린다. 일정에서 별도 큰 덩어리로 잡을 것. "기록 기능 1주" 식 뭉뚱그림 금지.
2. **카카오맵 역지오코딩 정확도 = 미검증 가정.** 좌표→장소명 정확도(특히 대형 건물·실내)는 붙여보기 전엔 모른다. **개발 착수 직후 가장 먼저 찔러볼 것.** 마지막에 확인했다 부정확하면 설계가 흔들린다.
3. **운영 부담의 슬금슬금 재발.** 사용자 문의·데이터 모니터링 같은 게 생기면 V1의 재판. MVP 범위(공유·탐색 제외)로 방어돼 있으나 일정에서도 지킬 것.

## 10. 미해결 이슈와 가정

### 10.1 가정 (검증 전제)

- **(가정 1) 타깃 부모 갤러리 사진의 GPS 생존율이 자동의 가치를 느낄 만큼 충분하다.** 운영자가 "참(대부분 있음)"으로 확정. 셀프 베타에서 재확인. 만약 대부분 비어 있으면 제품 무게중심이 "수동 입력 주, 자동 보너스"로 바뀜.
- **(가정 2) 카카오맵 역지오코딩이 좌표를 쓸 만한 장소 후보로 바꿔준다.** 미검증. 개발 착수 직후 최우선 검증(§8.1-2).
- **(가정 3) 소셜 루프 없이 내재 동기만으로 사용자가 돌아온다.** 제품의 핵심 베팅. 셀프 베타 + 첫 코호트로 검증.
- **(가정 4) 사용자가 앱의 넛지 없이 스스로 지도를 다시 연다(되새김).** 가장 얇은 가정. 순수 되새김 세션 지표로 감시.
- **(가정 5) 기록은 "회상하며 올리고 싶은 순간"에 일어나며, MVP엔 이를 유발할 장치가 없다.** 셀프 베타 1순위 관찰.
- **(가정 6) "1계정 = 1가족"으로 본다.** 개인 계정이지만 지표는 가족 단위로 센다. 부부가 같이 보려면 계정을 공유(같은 ID/비번)하며, 이는 제품이 만드는 기능이 아니다.

### 10.2 셀프 베타에서 확정할 열린 값

- 측정 창 (60일 / 30일)
- 계곡 통과 임계치 N (가설 10~20)
- 기존 장소 재방문 중복이 실제로 거슬리는 정도 → P1 승격 여부
- 로그인 전 입력 보존 구현 가능성 → 불가 시 F-8 (가) 폴백
