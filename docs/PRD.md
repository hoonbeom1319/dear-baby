# Dear Baby — PRD (Product Requirements Document)

> **버전:** v1.0 (MVP)
> **작성일:** 2026년 5월
> **상태:** 개발 착수 가능 (7단계 통과)

---

## 0. 이 문서를 읽는 법

이 문서는 1인 풀스택 개발자가 _바쁜 육아 중_ 자투리 시간으로 만드는 MVP의 PRD입니다. 다음 원칙으로 작성되었습니다.

- **기능은 최소로, 구조는 확장 가능하게.** 모든 결정이 이 원칙을 통과했습니다.
- **결정에는 근거가 따라옵니다.** 6개월 뒤 _"왜 이렇게 정했지?"_ 떠올릴 때 답을 주기 위해 근거를 같이 적었습니다.
- **빠진 것도 명시합니다.** Out of Scope 섹션이 진짜 PRD의 절반입니다. 무엇을 *안 하는지*가 무엇을 하는지만큼 중요합니다.

## 1. 배경과 문제 정의

### 1.1 배경

작년에 아이가 태어났고, 4개월쯤 외출이 가능해진 시점부터 *아이와 함께 갈 만한 장소*를 찾는 일이 시작됨. 그런데 _수유실, 기저귀 갈이대, 유모차 진입, 아기의자_ 같은 어메니티의 유무가 외출 가능 여부를 결정함. 가고 싶은 카페를 찾아도 기저귀 갈이대가 없으면 갈 수 없음.

기존 도구(네이버 지도, 카카오맵, 블로그, 맘카페)로 이 정보를 모으려면 **검색 → 후기 검색 → 광고 거르기 → 부모 후기 확인**의 여러 단계를 거쳐야 함. 바쁜 육아 중에 이걸 매 외출마다 반복하는 게 큰 부담.

### 1.2 핵심 문제

> 근처에 갈 만한 곳은 많다. 진짜 문제는 _"지금 우리 아이 월령으로, 부모도 편하게"_ 갈 수 있는 곳을 가려내는 일이고, 이걸 위해 여러 도구를 오가느라 시간이 너무 든다.

핵심은 **정보의 부재가 아니라 정보의 흩어짐**. 그리고 흩어진 정보 중 **광고가 아닌 진짜 정보**를 찾기가 어려움.

### 1.3 앱의 정체성

- **월령 기준 + 편의시설 정보를 신뢰성 있게 보여주는 큐레이션 앱**
- **판정하지 않음.** 시스템이 "갈 수 있다/없다"를 자동 판정하지 않고, 부모가 스스로 판단할 수 있도록 정보만 명확하게 제공
- **좁고 깊게.** 전국 얕은 데이터가 아니라 *특정 지역 깊은 데이터*로 시작

## 2. 목표와 성공 지표

### 2.1 진짜 목표

> 아이와 갈 곳을 탐색하는 시간을 _30분에서 5분으로_. "헤매지 않고 빠르게 결정."

### 2.2 측정 가능한 지표

**핵심 지표 (이 앱이 일을 하고 있는가)**

- **방문 → 행동 전환율**: 앱을 연 사용자 중 *길찾기 연결 또는 주소 복사*까지 도달한 비율
    - 목표 초안: **40% 이상**
    - 근거: "탐색 시간 단축"의 가장 정직한 대리지표. 들어와서 헤매다 나간 게 아니라 *정하고 떠났다*는 신호
- **세션 길이**: _짧을수록 성공._ 이 앱의 본질은 빠른 결정이라 세션이 짧다는 것은 마찰이 적다는 뜻

**성장 지표 (서비스로 자라고 있는가)**

- **주간 재방문율**: 이번 주 사용자가 다음 주에도 오는 비율
    - 목표 초안: **20% 이상**
    - 근거: 외출은 주 단위 행동이라 _재방문은 주 기준_
- **W4 잔존율**: 첫 사용 후 4주 차에도 남아 있는 사용자 비율

**건강 지표 (서비스의 토대가 멀쩡한가)**

- **장소 데이터 커버리지**: 타깃 동네에서 "갈 만한 곳" 중 등록 비율 (체감 기준)
- **정보 정확도**: 사용자 제보로 들어오는 "정보가 틀렸어요" 신고 건수 및 처리 속도 (목표: 48시간 내 대응)

### 2.3 지표 운영 원칙

- 위 숫자들은 **기준선**이지 **정답**이 아님. 운영하며 조정.
- _측정할 수 없는 목표_("30분 → 5분")가 아니라 *행동의 흔적*으로 환산.

## 3. 타깃 사용자

### 3.1 핵심 사용자

**1번 사용자 (MVP 우선 사용자): 돌 전(0~12개월) 아이를 키우는 부모**

근거:

1. **고통의 크기.** 돌 전엔 헛걸음의 대가가 가장 큼. 짐 잔뜩 싸서 갔는데 기저귀 갈이대가 없으면 그날 외출이 망가짐
2. **대체재의 부재.** 돌 이후 "키카+식당" 조합은 네이버 지도/키카 앱이 어느 정도 커버. *돌 전 편의시설 정보*는 빈틈
3. **사용자는 자라서 넘어옴.** 돌 전 부모를 잡으면 1년 뒤 자연스럽게 돌 이후 사용자가 됨

**2번 사용자 (MVP 동시 고려, 화면 설계 우선순위 낮음): 돌 이후 ~ 36개월 아이 부모**

키카+식당 조합 위주 사용. MVP 코스 기능이 이들을 부분적으로 받음.

### 3.2 사용 맥락

- **주로 집에서 외출을 *미리 계획*하는 순간.** 평일 저녁에 후보를 살피고, 외출 직전이나 며칠 전에 결정.
- **시간·체력 부족.** 한 손으로, 몇 분 안에 결정해야 함.
- **정보 정확성에 민감.** 헛걸음의 대가가 큼.

### 3.3 핵심 사용 시나리오

**시나리오 A — 평일 저녁, 미리 계획**
평일 밤 아이 재운 뒤, 주말 외출 후보를 둘러보고 즐겨찾기에 담음.

**시나리오 B — 주말 또는 외출 당일, 결정**
외출 당일 또는 주말 아침, 즐겨찾기에서 후보를 비교한 뒤 한 곳을 골라 길찾기까지 진행.

**시나리오 C — 링크 공유 받음**
배우자나 친구가 보내준 URL로 특정 장소·코스에 직접 진입. 앱 설치 없이 웹으로 바로 확인.

### 4.3 기능 명세 (상세 동작)

#### F-1. 장소 탐색

- 사용자가 앱을 열면 → *마지막으로 본 동네*의 장소 목록을 카드 형태로 표시
- 처음 사용자에겐 *운영자 지정 기본 동네*로 시작
- 카드는 *운영자가 정한 표시 순서*로 정렬
- 사용자가 카드를 탭하면 → 장소 상세 화면으로 이동

#### F-2. 동네 칩

- 첫 화면 상단에 운영 중인 동네들이 칩으로 표시 (가로 스크롤, 자동 재생 없음)
- 현재 선택된 동네는 시각적으로 구분
- 다른 칩 탭 → 그 동네 장소 목록으로 즉시 전환
- 동네가 많아져 한 줄에 안 들어가면 _우측 시각 힌트_(절반 보이는 칩 또는 그라데이션)로 더 있음을 알림

#### F-3. 카테고리 칩

- 동네 칩 바로 아래에 위치
- `[전체] [카페] [식당] [키즈카페] [복합몰]` 형태. 단일 선택
- "전체"가 기본 선택. 다른 칩 탭 → 그 카테고리만 필터
- 카테고리 = "전체"일 때만 _추천 코스 진입 카드_ 노출. 다른 카테고리 선택 시 코스 카드 숨김

#### F-4. 코스 진입 카드

- 카테고리 = "전체"일 때, 카테고리 칩 아래 한 줄짜리 진입 카드 노출
- 텍스트 예: _"일산의 추천 코스 보기 →"_
- 탭 → 코스 목록 페이지

#### F-5. 장소 카드

표시 정보:

- 장소 이름
- 카테고리 라벨 + 동네 라벨 (예: "카페 · 일산")
- 권장 월령 범위 (예: "6개월~3살")
- 동네 내 위치 (예: "정발산동")
- 한 줄 설명/큐레이터 코멘트
- 사진 (있을 때만, 없으면 자리 비움 — 기본 이미지 깔지 않음)
- 있는 편의시설만 컬러 아이콘 (없는 것은 카드에 표시하지 않음)

#### F-6. 장소 상세 화면

**구조 (위에서 아래로):**

1. **상단 헤더**
    - 뒤로가기 (앱 내 히스토리 있으면 직전 화면, 없으면 _홈으로_)
    - 우측: 별(즐겨찾기), 공유 아이콘

2. **핵심 정보 블록**
    - 장소 이름 (큰 글씨)
    - 카테고리 · 동네 라벨
    - 권장 월령 범위
    - 한 줄 설명/큐레이터 코멘트

3. **사진 영역** (있을 때만)
    - 1~3장 슬라이드 (좌우 스와이프). 캡션 없음

4. **편의시설 전체 표시**
    - 운영 중인 _모든_ 편의시설 항목 나열
    - 그 장소가 가진 것 = 컬러/체크
    - 그 장소가 안 가진 것 = 흑백/빈 표시
    - "한눈에 무엇이 있고 없는지 비교" 가능

5. **정보 제보**
    - _"이 정보가 맞나요?"_ 안내 + [👍 맞아요] [🙅 바뀌었어요] 두 버튼
    - 🙅 탭 → 무엇이 바뀌었는지 객관식 후속 선택 → Report 한 줄 생성
    - 비회원도 가능 (User 연결은 선택)

6. **기본 정보**
    - 주소 + 옆에 복사 아이콘
    - 전화번호 + 옆에 전화 아이콘
    - (영업시간/휴무 없음 — Out of Scope 참고)

7. **하단 액션 영역 (한 손 영역)**
    - **큰 길찾기 버튼** + 옆에 작은 아이콘 3개 (주소복사, 전화, 즐겨찾기)
    - 영역 6의 아이콘과 중복되지만, *읽다가 누르는 자리*와 *결정한 뒤 누르는 자리*의 맥락이 다르므로 양쪽에 둠
    - 길찾기 탭 → 시트 띄움: `[카카오내비] [티맵] [네이버지도] [주소복사]`

#### F-7. 즐겨찾기 (Favorite)

- 별 아이콘 탭 → 비회원이면 로그인 시트, 로그인 사용자면 즉시 등록/해제
- 비회원 로그인 흐름: 별 탭 → 로그인 시트 → 카카오/네이버 → **원래 보던 화면 정확히 복귀 + 자동 등록**
- 즐겨찾기 화면 진입: 첫 화면 우측 상단 별 아이콘 (다른 진입점 없음 — 일관된 사용자 경험)

#### F-8. 즐겨찾기 목록 화면

- 헤더: "즐겨찾기" 타이틀 + 뒤로가기
- 본체: 첫 화면과 _같은 카드 디자인 재사용_ (별이 채워진 상태)
- 별 탭 시 즐겨찾기 해제
- 정렬: 최근 추가순 자동 (사용자 선택 불가)
- 필터/칩 없음
- 빈 상태: 안내 메시지 + 첫 화면 입구

#### F-9. 코스 목록 페이지

- 헤더: 동네 이름 + 뒤로가기
- 그 동네의 코스들을 카드 형태로 나열
- 탭 → 코스 상세

#### F-10. 코스 상세 화면

**구조:**

1. **상단 헤더**
    - 뒤로가기 (히스토리 없으면 _홈으로_)
    - 우측: 공유 아이콘

2. **핵심 정보**
    - 코스 이름 (큰 글씨)
    - 소속 동네 라벨
    - 예상 소요 시간 (있을 때)
    - 코스 설명

3. **정거장 목록**
    - CourseStop들을 _순서대로_ 나열
    - 각 정거장: 순서 번호 ①②③ + 장소 카드 (재사용) + 큐레이터 짧은 코멘트
    - 정거장 사이는 *선 또는 화살표*로 흐름 표시 (즐겨찾기와 시각적 구분)
    - 카드 탭 → 장소 상세

4. **코스 끝 표시** (선택)
    - 마지막 정거장 아래 짧은 "코스 완료" 또는 끝 라벨

5. **하단 영역**
    - 큰 액션 없음
    - 결정은 각 장소 상세에서 일어남
    - 공유는 상단 우측 아이콘으로 충분

#### F-11. 장소 상세의 "함께 가면 좋은 코스"

- 장소 상세 어딘가에 이 장소가 포함된 코스 1~2개를 카드로 표시
- 탭 → 코스 상세
- _우연한 발견_ 경로

#### F-12. 로그인 시트

- 화면 하단에서 슥 올라오는 시트 (별도 페이지 아님)
- 짧은 안내 문구 + [카카오 시작하기] [네이버 시작하기] (카카오 위, 네이버 아래)
- 마지막 로그인 수단이 있으면 그 버튼에 _"최근 로그인"_ 작은 라벨 (위치는 안 바뀜)
- 닫기/돌아가기 출구 명확
- 가입/로그인 통합 단일 버튼 (소셜 OAuth가 자동 처리)
- 첫 가입 시 약관 동의 화면 한 번 (서비스 이용약관 + 개인정보 처리방침)
- 닉네임은 소셜에서 자동, 사용자 입력 없음

#### F-13. 정보 제보 (Report)

- 비회원도 가능
- 🙅 선택 시 객관식: "기저귀 갈이대 없어졌어요", "수유실 없어졌어요" 등 (편의시설 항목별)
- 생성된 Report는 *관리자 대시보드*로 들어가 대기 상태
- 관리자가 _반영_ 누르면 한 번 더 확인 후 데이터 변경

#### F-14. 링크 공유

- 장소 상세, 코스 상세에서 공유 아이콘 탭 → 현재 화면의 URL을 클립보드에 복사
- URL은 쿼리에 장소/코스 ID를 포함, 받은 사람이 열면 해당 화면이 직접 렌더링
- 웹앱이라 받은 사람이 앱 설치 없이 브라우저에서 바로 진입 가능

## 5. 관리자 페이지 요구사항

### A-1. 관리자 인증

- 사용자 소셜 로그인 재사용 (카카오/네이버)
- User 엔터티의 _관리자 여부_ 속성으로 권한 식별

### A-2. 대시보드

- 새 제보 N건 알림 (탭 → 제보 관리)
- 장소 통계 (동네별 등록 개수)
- 빠른 진입 버튼 [장소 추가] [코스 추가]

### A-3. 장소 관리

- 목록 화면: 테이블 (이름 / 동네 / 카테고리 / 권장 월령 / 편의시설 개수 / 수정 / 삭제)
- 동네별·카테고리별 필터, 등록일/이름 정렬
- [+ 새 장소 추가] 버튼
- 추가/수정 폼 입력 필드:
    - 이름, 동네(드롭다운), 카테고리(드롭다운), 주소(텍스트), 전화번호
    - 권장 시작/끝 월령 (개월 단위 숫자)
    - 한 줄 설명
    - 편의시설 (체크박스 다중 선택)
    - 사진 (선택, 1~3장)
    - 표시 순서 (숫자)
- 주소 입력 시 _네이버 지도 지오코딩으로 위도·경도 자동 채움_ (운영자는 좌표 입력 불필요)
- 동네·카테고리 드롭다운에 _"+ 새 항목 추가" 인라인 옵션_ (별도 페이지 이동 없이 추가 가능)

### A-4. 코스 관리

- 목록 화면: 테이블 (이름 / 동네 / 정거장 수 / 수정 / 삭제)
- 동네별 필터
- 추가/수정 폼:
    - 이름, 동네(드롭다운), 설명, 예상 소요 시간(선택)
    - 정거장 목록: [+ 정거장 추가] → 장소 검색·선택 → 추가
    - 각 정거장에 짧은 코멘트 입력 칸
    - 순서 변경: ↑ ↓ 버튼

### A-5. 제보 관리

- 목록: 제보 시각 / 장소 / 유형(👍/🙅) / 변경 내용 / 상태(대기/반영/무시)
- 상태별 필터 (기본: 대기 중)
- [반영] 버튼 → 확인 모달 → 데이터 자동 수정
- [무시] 버튼 → 상태만 변경, 데이터 유지

### A-6. 보조 영역

- 동네 목록 관리 (이름, 표시 순서)
- 카테고리 목록 관리
- 편의시설 목록 관리

## 6. 데이터 모델 (개념)

> **범위 명시:** 이 섹션은 *개념 모델*이며, 물리 스키마(컬럼 타입, 인덱스, 정규화, 마이그레이션 등)는 개발 단계에서 확정.

### 6.1 엔터티 목록

```
[사용자 User]
  - 사용자 ID
  - 소셜 로그인 정보 (카카오/네이버 식별자)
  - 닉네임 (소셜에서 자동)
  - 관리자 여부
  - 가입 시각

[동네 Area]
  - 동네 ID
  - 동네 이름
  - 표시 순서 (선택)

[카테고리 Category]
  - 카테고리 ID
  - 이름 (카페, 식당, 키즈카페, 복합몰)
  - 아이콘/표시 정보
  - 표시 순서 (선택)

[편의시설 Amenity]
  - 편의시설 ID
  - 이름 (수유실, 기저귀 갈이대, 아기의자, 유모차 접근, 주차 가능 등)
  - 아이콘 정보

[장소 Place]
  - 장소 ID
  - 이름
  - 주소
  - 위도, 경도 (주소 입력 시 자동 채움)
  - 전화번호
  - 권장 시작 월령 (개월 단위)
  - 권장 끝 월령 (개월 단위)
  - 한 줄 설명 / 큐레이터 코멘트
  - 사진 (0~3장)
  - 표시 순서 (목록 정렬용)
  - 소속 동네 (Area 참조)
  - 소속 카테고리 (Category 참조)

[코스 Course]
  - 코스 ID
  - 이름
  - 설명
  - 예상 소요 시간 (선택)
  - 소속 동네 (Area 참조)

[코스장소 CourseStop]
  - 코스장소 ID
  - 어느 코스 (Course 참조)
  - 어느 장소 (Place 참조)
  - 순서 (1, 2, 3...)
  - 짧은 코멘트

[즐겨찾기 Favorite]
  - 즐겨찾기 ID
  - 어느 사용자 (User 참조)
  - 어느 장소 (Place 참조)
  - 저장 시각

[정보제보 Report]
  - 제보 ID
  - 어느 장소 (Place 참조)
  - 누가 (User 참조, 비회원이면 비어 있음)
  - 제보 유형 (맞아요 👍 / 바뀌었어요 🙅)
  - 무엇이 바뀌었나 (객관식)
  - 제보 시각
  - 관리자 처리 상태 (대기/반영/무시)

[장소-편의시설 연결]  (N:M 관계 표현)
  - 어느 장소 (Place 참조)
  - 어느 편의시설 (Amenity 참조)
  - 연결 자체가 "있음"의 의미
```

### 6.2 관계 한눈에

```
Area     1 ─── N  Place
Area     1 ─── N  Course
Category 1 ─── N  Place
Place    N ─── M  Amenity      (장소-편의시설 연결)
Course   N ─── M  Place        (중간: CourseStop, 순서·코멘트)
User     N ─── M  Place        (중간: Favorite)
User     N ─── M  Place        (중간: Report, User는 선택)
```

### 6.3 설계 원칙

- **편의시설은 별도 엔터티 (N:M)**: 새 편의시설 추가가 장소 구조를 안 건드림 → 확장성
- **동네·카테고리는 1:N**: 가장자리 케이스(여러 동네 걸친 장소 등) 무시하고 단순함 우선 — 운영 혼선 최소화
- **다중 카테고리 미지원**: "복합몰"이라는 카테고리로 스타필드 같은 케이스 해결
- **즐겨찾기·정보제보는 중간 엔터티**: User와 Place의 N:M 관계를 명시적으로 표현

## 7. 사용자 흐름

### 7.1 시나리오 A — 평일 저녁, 미리 계획

```
앱 열기
  → [첫 화면] 기본/마지막 동네 장소 목록
  → 동네 칩으로 다른 동네 전환 (선택)
  → 카테고리 칩으로 카테고리 필터 (선택)
  → 장소 카드 탭
  → [장소 상세] 영역 1~7 스크롤
  → 별 탭 (즐겨찾기)
  → [로그인 시트] (비회원이면)
  → 카카오/네이버 로그인 + 약관 동의 (첫 사용 시)
  → [장소 상세] 복귀 + 즐겨찾기 자동 등록
  → 뒤로가기 → [첫 화면] 다음 후보 탐색 반복
```

### 7.2 시나리오 B — 결정과 길찾기

```
앱 열기
  → [첫 화면] 우측 상단 별 탭
  → [즐겨찾기 목록] 최근 추가순
  → 카드 비교, 후보 선택, 탭
  → [장소 상세]
  → 하단 [길찾기] 탭
  → 시트: [카카오내비] [티맵] [네이버지도] [주소복사]
  → 선택한 앱으로 딥링크 (좌표 + 장소 이름 함께 전송)
```

### 7.3 시나리오 B 변형 — 코스 경로

```
앱 열기
  → [첫 화면] 코스 진입 카드 탭
  → [코스 목록]
  → 코스 카드 탭
  → [코스 상세] 정거장 순서대로 확인
  → 특정 정거장 카드 탭 → [장소 상세] → 길찾기
  → 또는 → [장소 상세]의 "함께 가면 좋은 코스"로 다른 코스 발견
```

### 7.4 시나리오 C — 링크 공유 받음

```
와이프가 카톡으로 받은 URL 탭
  → 브라우저에서 [장소 상세] 또는 [코스 상세] 직접 진입
  → 화면 자체로 모든 정보 파악 가능 ("홀로 서는 화면" 원칙)
  → 뒤로가기 = "홈으로" (히스토리 없으므로)
  → 별/제보/길찾기 등 모든 기능 동일 작동
```

### 7.5 로그인 후 복귀 흐름

```
비회원이 별 탭
  → [로그인 시트]
  → 카카오/네이버 OAuth
  → 콜백 후 *원래 보던 장소 상세로 정확히 복귀*
  → 별 자동 채움 (즐겨찾기 자동 등록)
```

이 복귀 흐름은 별도 화면이 아니라 _모든 로그인 유도 지점에 공통 적용되는 규칙._

### 8.2 관리자 화면 (5영역)

| #   | 화면          | 핵심 역할               |
| --- | ------------- | ----------------------- |
| 1   | 관리자 로그인 | 소셜 로그인 + 권한 확인 |

### 8.3 화면 간 이동 규칙

- **로그인 후 복귀**: 항상 *로그인 직전에 보던 화면*으로 정확히 복귀
- **카드 디자인 재사용**: 첫 화면, 즐겨찾기 목록, 코스 상세의 정거장 모두 같은 카드 컴포넌트 사용 → 학습 비용 최소화

## 9. 디자인 방향

### 9.1 톤

- **신뢰감 있고 차분함.** 정보의 정확성이 핵심 가치이므로 _과한 장식·화려한 컬러_ 지양
- **육아 중 사용자가 편하게.** 부드럽고 따뜻한 분위기, 피로한 눈에 자극적이지 않은 컬러 팔레트
- **큐레이션 색깔.** 운영자가 직접 고른 곳들이라는 *손맛*이 느껴지게

### 9.2 레퍼런스 서비스 (2~3개)

- **카카오맵 / 네이버 지도**: 정보 밀도와 가독성 참고 (단, 이 앱은 더 단순하고 큐레이션 색깔이 진해야 함)
- **오늘의집**: 카드 기반 콘텐츠 탐색의 부드러운 톤
- **(추후 디자인 단계에서 후보 1~2개 추가)**

### 9.3 디바이스 우선순위

- **모바일 웹 우선.** 사용자 대부분이 폰으로 사용
- **데스크톱은 관리자 페이지만.** 사용자 화면은 데스크톱 최적화 우선순위 낮음 (반응형으로 가능한 정도까지)

### 9.4 한 손 조작

- 핵심 액션은 *화면 하단 엄지 닿는 위치*에 배치
- 자주 누르는 것 = 하단, 가끔 누르는 것 = 상단 원칙

### 9.5 다크모드

- **MVP 미지원.** 출시 후 사용자 요청 모이면 검토

## 10. 비기능 요구사항

### 10.1 성능

- 첫 화면 로딩: 모바일 4G 환경에서 _3초 이내_ 첫 의미 있는 콘텐츠 표시
- 장소 상세 진입: 1초 이내
- 카드 스크롤: 60fps 부드러움

### 10.2 가용성 / 신뢰성

- 외부 API(카카오/네이버 OAuth, 지도 API) 실패 시 _명확한 에러 안내_
- 내비 딥링크가 해당 앱 미설치로 실패할 경우 _주소 복사로 대체 안내_

### 10.3 데이터 무결성

- 정보 제보의 [반영] 시 한 번 확인 모달 → 운영자 실수 방지

### 10.4 보안 / 프라이버시

- 사용자 데이터 최소 수집 (소셜에서 받은 식별자 + 닉네임만)

- 약관·개인정보 처리방침 출시 전 게시 필수

### 10.5 호환성

- 최신 모바일 브라우저(Chrome, Safari, Samsung Internet) 우선 지원
- iOS Safari, Android Chrome 우선 테스트

### 10.6 접근성

- 핵심 액션 버튼 크기: _최소 44×44pt_ (Apple HIG 기준)
- 색상 대비: WCAG AA 수준 준수 (편의시설 컬러/흑백 구분도 충족)

## 11. 범위 외 (Out of Scope)

### 11.1 데이터 항목 (의도적으로 _수집·표시하지 않음_)

- **영업시간 / 휴무일**: 운영자가 정확하게 유지보수 불가능. 부정확한 정보는 앱 신뢰도를 깎으므로 _제공하지 않음_. 사용자는 전화 걸기로 확인
- **노키즈존 정보**: 이 앱의 정체성("아이랑 갈 수 있는 곳")과 어긋남. 노키즈존은 _애초에 등록 대상이 아님_

### 11.2 기능 (Phase 2 이후)

- 편의시설 필터, 지역 필터, 월령 필터
- 자유서술 후기, 별점
- NEW 뱃지, 즐겨찾기 수 뱃지
- 키워드 검색창
- 지도 뷰
- 푸시 알림
- 코스 자동 생성
- GPS 기반 즉석 추천
- 다크모드
- 코스 단위 즐겨찾기

### 11.3 인프라

- 네이티브 앱 (iOS/Android) — 웹앱으로 시작
- 다국어 지원 — 한국어만
- 대규모 트래픽 대응 — 초기엔 Vercel + Supabase 무료 한도로 운영

### 11.4 데이터 범위

- **MVP는 동네 2곳: 서울 송파권, 파주 운정권**
- 그 외 지역은 데이터 축적 후 점진 확장
- 출시 기준: *각 동네 25~30개 장소*가 채워진 시점

## 12. 가정과 미해결 이슈

### 12.1 가정 (검증되지 않았지만 전제로 두는 것)

- **A1.** 1번 사용자(돌 전 부모)가 가장 큰 헛걸음 고통을 겪고 있다 → 출시 후 실제 사용 패턴으로 검증 필요
- **A2.** "30분 → 5분"의 목표는 *길찾기/주소복사 도달률 40%+*로 환산 가능하다 → 출시 후 실제 수치와 대비
- **A3.** 사용자는 자유서술 후기를 _읽고 싶어 하지만 쓰지는 않을 것이다_ → 출시 후 정보 제보 활용도로 간접 검증
- **A4.** 영구 즐겨찾기 + 소셜 로그인의 마찰을 사용자가 감수할 것이다 → 가입 전환율과 즐겨찾기 사용률로 검증
- **A5.** 운영자가 동네 2곳에 각 25~30개 장소를 _3~5개월 내_ 채울 수 있다 → 데이터 채우기 일정 모니터링

### 12.2 미해결 이슈 (개발 단계 또는 출시 후 결정)

- **I1.** Supabase Auth로 카카오/네이버 연동을 처리할지, 직접 OAuth 구현할지 — 개발 착수 시점에 셋업 비교 후 결정
- **I2.** 사진 저장 용량 한도 (Supabase Storage 무료 한도) 모니터링 정책
- **I3.** 정보 제보 악용·장난 제보 대응 — 비회원 허용으로 시작하되, 감당 안 되면 로그인 필수로 전환 검토
- **I4.** 동네 데이터 충분도의 *체감 기준*은 운영자 본인 + 베타 사용자 의견으로 결정
- **I5.** 디자인 시안 — PRD 외 별도 단계로 진행

### 13.2 가용 시간 및 일정

- **가용 시간:** 주 5시간 이상 (고정 블록으로 운영)
- **개발 가속:** Claude Code / Cursor 활용
    - 가속됨: 코드 작성, 보일러플레이트
    - 가속 안 됨: 데이터 채우기, 외부 디버깅, UX 결정, 호스팅 설정

### 13.4 기술 스택

- **호스팅:** Vercel

- **지오코딩:** 네이버 지도 API (주소 → 위경도)
- **내비 딥링크:** 카카오내비, 티맵, 네이버 지도
- **소셜 로그인:** 카카오 OAuth, 네이버 OAuth

## 15. 부록

### 15.1 5단계에서 박은 메모 (개발 시 챙길 것)

1. 장소·코스 상세는 *링크로 직접 진입한 사람*도 만족하도록 설계 (홀로 서는 화면 + 홈으로 입구)
2. 딥링크 URL은 _좌표 + 장소 이름_ 둘 다 실어 보냄
3. 길찾기 시트: 카카오내비 / 티맵 / 네이버 지도 / 주소복사 — 4개 옵션
4. 카테고리 칩 + 동네 칩은 같은 상단 영역에서 짝으로 배치
5. 카드는 *필터 없이 훑기*가 핵심이므로 정보 한눈에 들어와야 함
6. 로그인 후 _원래 화면 정확히 복귀_ — 보던 장소의 별이 자동 채워진 상태로
7. 관리자 제보 처리: 자동 반영 금지, 운영자가 검토 후 확인 모달로 결정
8. 관리자 장소 등록: 동네/카테고리는 드롭다운(인라인 추가 가능), 편의시설은 체크박스, 주소 입력 시 좌표 자동 채움

### 15.2 용어 정리

- **MVP**: Minimum Viable Product. 최소 기능으로 출시해 학습하기 위한 버전
- **운영자**: 이 앱의 유일한 관리자. 데이터를 직접 채우고 제보를 처리하는 사람
- **큐레이션**: 운영자가 직접 고르고 다듬은 콘텐츠. 자동 알고리즘이 아닌 사람의 손맛
- **편의시설 (Amenity)**: 부모에게 _있으면 좋은_ 시설. 부정적 속성(노키즈존 등)은 포함하지 않음
- **정거장 (CourseStop)**: 코스 안의 한 장소. 순서와 큐레이터 코멘트를 가짐
